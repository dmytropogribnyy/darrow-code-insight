## 7 · CONCEPTUAL SECTION FIELD MODEL

Conceptual fields only — no code, no types, no validation syntax.

**Normal generated section may carry:**

- `opening_line` — short declarative hook
- `scenario` — lived recognizable moment (where required)
- `prose` — body paragraphs
- `key_insight` — one emphasized line (optional, sparse)
- `protocols[]` — behavioral callouts (title + body)
- `warning_signals[]` — observable early-cue callouts
- `proof_tags[]` — quiet data confirmation (≤5)

**Special fields:**

- `disclaimer` — vitality_baseline only (verbatim medical disclaimer)
- `before_after_pairs` — before_after only (exactly 2 Before / 2 After)
- `executive_summary_blocks` — executive_summary only (the 6 labeled blocks)
- `closing_pillars` — next_step only (4 pillars, frame fixed / conclusions variable)
- `cover_tagline` — cover sub-field, not a body section

Which sections carry protocols/warnings is defined in the module spec (§8 there)
and the AI prompt (§19 there). This document records the conceptual shape only;
exact field types and minimums are decided during implementation.

## 8 · FIELD RESPONSIBILITY RULE

- The **AI generates content** (the words).
- The **template controls layout and visual hierarchy** (the look).
- The AI must **not** decide visual formatting through raw/random markdown.

Content and presentation are separated so every client's report renders
consistently and layout never drifts from inconsistent AI formatting.

---

## 9 · TEMPLATE RENDERING RESPONSIBILITIES

The template (conceptually) renders:

- section title
- opening line
- scenario (visually distinct)
- prose paragraphs (short, with breathing room)
- key insight (sparse emphasis)
- protocol callouts (lifted from prose)
- warning signal callouts (muted, quieter than protocols)
- proof tags (small, muted, at the end)
- disclaimer (vitality_baseline)
- before/after blocks (scannable Before:/After: labels)
- executive summary labels (the 6 sublabels)
- closing pillars (the 4 pillars)
- all static pages (cover, orientation system, library, back cover)

The template owns all of the above visual treatment. The AI provides only the
structured content fields it renders from.

## 10 · VALIDATION RULES — CONCEPTUAL ONLY

Stated as intent, not as Zod/TypeScript. The migration will implement these as
actual validation later.

- all 17 required section keys present
- no extra/unknown section keys
- per-section word count within target range (warning, not hard fail, on minor
  drift; flag significant under/over-shoot)
- `proof_tags` ≤ 5 per section
- proof tags reference only available data
- required `disclaimer` present in vitality_baseline
- before_after contains exactly 2 Before / 2 After pairs
- protocols present where the master pattern / module spec requires them
- warning signals present where required
- no city recommendations anywhere in CORE
- no direct cross-sell inside body sections
- if `birth_time_known=false`: no houses, angles, Ascendant, MC, IC, or Descendant
- if BaZi unavailable: no BaZi claims or tags
- if `full_name_for_numerology` absent: no Expression / Soul Urge / Personality
- if transits unavailable: no transit claims

Do not write Zod or TypeScript for these here. This is the conceptual checklist
the implementation will encode.

## 11 · QUALITY GATE RULES

Conceptual checks the generation should pass (drawn from the content standard's
pass/fail checklist):

- scenario-first (no section opens with chart data)
- human-readable (a non-astrologer can follow it)
- no chart dump
- no old v3 targets (e.g. 18–20 pages / 3,000–3,600 words)
- no promises, diagnosis, or prediction
- no mechanical "your system" overuse
- no product-teaser language
- no PLACE city claims in CORE

A section failing any quality gate is regenerated, independent of whether it
passes structural validation.

## 18 · FAILURE BEHAVIOR

v4.1 generation fails if it:

- is missing any required section key
- uses unavailable data
- exceeds 5 proof tags in a section
- includes city recommendations in CORE
- uses old v3 targets as the target
- mismatches the product naming against checkout (must read CORE Report: UNVEIL)
- includes a direct cross-sell inside a body section

A failed generation is rejected and regenerated — never silently shipped, never
silently downgraded to v3 keys.
