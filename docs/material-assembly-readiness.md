# Material Assembly Readiness

**Phase:** DATA-AUDIT-1 (docs + diagnostic tooling) · **Updated:** 2026-06-06 · **Run mode:** plan-only

Can the system, for **every new paid report**, reliably assemble the full material pack
(raw data + derived data + interpretive dictionaries + availability flags + fallbacks +
safety constraints) **without inventing unsupported material**? This doc answers that.

> **Status of this audit:** the FreeAstroAPI availability run is **plan-only** — `FREEASTROAPI_KEY`
> is not available locally, so no approved provider run was executed. Field reliability marked
> *unverified* stays unverified until an approved run:
> `FREEASTROAPI_AUDIT_APPROVE=1 ASTRO_PROVIDER=freeastroapi npm run audit:freeastroapi`.

> ⚠️ **Headline finding:** the only **coded** interpretive dictionary is
> `src/lib/numerology/numerology-meanings.ts`. All other interpretation
> (signs / planets / houses / aspects / elements / BaZi / archetypes / colors / stones) is
> generated by the **AI from training, guided by prompt rules** — or exists **doc-only** in
> `docs/knowledge/` (not wired to code/prompts). There is **no deterministic interpretive
> material layer** beyond numerology.

---

## 1 · What material exists today

- **Deterministic provider data** (FreeAstroAPI): natal, houses/angles (birth-time gated),
  aspects, transits, solar_return, moon_phase, bazi, bazi_flow — *code path exists, real-field
  reliability unverified*.
- **Derived/computed (internal):** Life Path, Birth Day Number, Personal Year, name numerology
  (when full name present) + their meanings (`numerology-meanings.ts`) — *implemented + verified*.
- **Availability flags:** `birth_time_source`, `bazi.available`, `hour_pillar_*`,
  `*.available/usable`, `time_confidence` — implemented.

## 2 · What material is only planned (doc-only, not wired)

`docs/knowledge/`: `ASTRO_INTERPRETATION_RULES_v1`, `BAZI_INTERPRETATION_RULES_v1`,
`ARCHETYPE_LIBRARY_v1`, `COLORS_STONES_SYMBOLIC_ALLIES_v1`, `SYMBOLIC_IDENTITY_STANDARD`,
`NUMEROLOGY_RULES_v1`, mapping/ modules/ rules/ source_packs/. These are **policy/structure
docs**, not deterministic dictionaries consumed by code or prompts.

## 3 · What material is missing

- Coded interpretive dictionaries for signs / planets / houses / aspects / elements / BaZi
  (Day Master, Five Elements, Ten Gods, luck cycles, Chinese zodiac).
- Approved color / stone / symbolic-ally dictionary (currently **gated off** in the v4 prompt).
- Any Japanese-astrology, astrocartography, or compatibility/synastry source.
- Per-module content contracts for add-ons (MODULE-PATTERN-1).

## 4 · What must NOT be claimed yet

Colors, stones/crystals, **Japanese astrology**, astrocartography / city guidance,
compatibility/synastry, medical/health diagnosis, financial guarantees, guaranteed predictions.
(See the `do_not_claim` / `not_implemented` rows in `material-readiness.ts`.)

## 5–11 · Safe material categories per module

Legend: ✅ safe today · ⚠️ allowed but unverified provider data · ⛔ must not claim.

| Module | ✅ Safe | ⚠️ Unverified | ⛔ Do not claim |
|---|---|---|---|
| **5. CORE** | date numerology + meanings; natal narrative (AI) | natal placements, BaZi, houses (birth-time) | colors/stones, Japanese astrology, guarantees |
| **6. LOVE** | Venus/Mars/Moon narrative (AI) | 5H/7H/Desc, name-num reinforcement | synastry/compatibility, name-only compatibility |

| **8. BODY** | "your system may respond to…" narrative | 6H, BaZi imbalance, moon-phase rhythm | medical/health diagnosis |
| **9. YEAR** | Personal Year theme | transits, solar return, bazi_flow | guaranteed predictions |
| **10. STYLE** | Venus/Moon visual-resonance (AI) | ASC, BaZi element balance | colors/stones (gated), healing/luck/protection |
| **11. PLACE** | Moon/environment qualities (AI) | IC/4H/angular, BaZi favorable elements | astrocartography / specific cities |

**All modules currently rely on AI-from-training for interpretation** (except numerology), so
"safe today" means *the data gating + safety rules are in place*, not that a deterministic Darrow
dictionary exists. Depth/accuracy parity per module is **not** yet achieved (MODULE-PATTERN-1/DIAG-1).

## 12 · Required before public launch of all modules

1. **DATA-AUDIT-1 approved run** — verify real FreeAstroAPI field availability across cases A–E.
2. **MODULE-PATTERN-1** — per-module content contracts (prompt/schema/length/safety).
3. **MODULE-DIAG-1** — real generation + visual PDF QA for each add-on.
4. **ANCHOR-AUDIT-1** — per-module proof/data validation (extend B5.3-A beyond CORE).
5. **BUNDLE-B/C** — separate PDFs per module (delivery).
6. Decide colors/stones: approve a verified dictionary **or** keep them out of copy & prompts.

## 13 · What to block from prompts until verified

Colors, stones, Japanese astrology, astrocartography/cities, compatibility/synastry, medical
claims, financial/predictive guarantees. The v4 prompt already gates colors/stones and
city/medical/guarantee claims — keep that, and do **not** add enrichment layers without a
verified source.

## 14 · Future Report Context Builder

**Does it exist?** No production context builder assembles a per-module material packet today —
the production path passes the whole `DarrowChartData` + one-line `moduleRouting()` hints to a
single AI call.

A **diagnostic-only** prototype is added here (not wired to production):
`src/lib/diagnostics/material-context/` →
`buildMaterialContextForModule(module, chart)` returns `{ availability, allowedFields,
forbiddenFields, dictionaries, fallbackRule, safetyGuardrails, forbiddenClaims, rules }`.

A future production `buildReportContextForModule(module, reportId)` should return: customer
basics, birth data, normalized chart, availability flags, module-allowed/forbidden fields,
proof anchors, the (future) interpretive dictionary pack, fallback rules, safety guardrails,
confidence notes, and the forbidden-claim list — enforcing **No invented enrichment** and
**No unsupported public claims**.

## Governing rules (enforced in tooling + tests)

- **No invented enrichment** — if a category is not backed by deterministic data, an approved
  dictionary, or an approved prompt contract, the AI must not present it as the Darrow method.
- **No unsupported public claims** — public/report copy must not imply colors, stones, Japanese
  astrology, astrocartography, compatibility, or medical/financial/guaranteed layers unless the
  source is implemented, verified, and approved.

## Tooling

- `npm run audit:freeastroapi` — plan-only (no network) by default; approved run probes
  FreeAstroAPI with **synthetic** inputs and writes sanitized summaries to gitignored
  `outputs/freeastroapi-audit/`.
- Pure modules: `availability.ts` (`summarizeAvailability`, `moduleImpact`),
  `material-readiness.ts` (`MATERIAL_CATEGORIES`, `MODULE_PACKS`, `buildMaterialContextForModule`).
