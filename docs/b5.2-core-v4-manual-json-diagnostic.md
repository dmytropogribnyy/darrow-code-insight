# B5.2 тАФ Manual CORE v4 JSON Diagnostic CLI

**Date:** 2026-06-04
**Status:** Complete. Manual/local only. **Production remains v3.**

---

## 1 ┬╖ Scope

A small, manual, local CLI that runs the **existing staged** CORE v4.1 path
end-to-end for diagnostics тАФ without touching production. It validates, in one run:

1. the staged v4 prompt (`darrowcode_ai_system_prompt_v4_1.md`)
2. `generateCoreV4Split()`
3. `CoreV4Schema`
4. `evaluateCoreV4Structure()`
5. `evaluateCoreV4Lengths()`
6. the v4 renderer (`renderCoreV4HtmlSafe`) + the single PDF engine

It is **not** a production switch, **not** a web route, **not** checkout/customer
integration. No existing generation logic is reimplemented тАФ the CLI orchestrates
the existing functions.

## 2 ┬╖ Files

| File | Role |
|---|---|
| `src/lib/ai/fixtures/core-v4-diagnostic-input.ts` | Deterministic diagnostic input (Dmitry; `birth_time_known=true`, full name, BaZi sex). Builds the chart via `MockAstroProvider` and the user prompt via `buildCoreV4UserPrompt`. **No FreeAstroAPI, no customer data.** |
| `src/lib/ai/diagnostics/core-v4-diagnostic.ts` | Pure logic: env-options, plan text, `assertCanRunAi` guard, `runCoreV4Validation` (schema + structure + lengths), `formatValidationReport`, and the render-only `buildCoreV4DiagnosticClientSnapshot`. |
| `src/lib/ai/diagnostics/run-core-v4-diagnostic.test.ts` | The CLI runner (Vitest spec). Plan-only by default; approved path gated. |
| `src/lib/ai/diagnostics/core-v4-diagnostic.test.ts` | Mocked tests (no real AI). |
| `scripts/generate-v4-pdf.mjs` | Generalized with optional `CORE_V4_PDF_IN` / `CORE_V4_PDF_OUT` env so the CLI reuses the **single** PDF engine. |

### Why Vitest (not a flag CLI)

`tsx` and `vite-node` are not installed, and the v4 generator/renderer use Vite-only
imports (`?raw`, `?inline`) that plain `node` cannot resolve. Per the repo's existing
convention (`generate:html`), the CLI runs through **Vitest**, which provides the Vite
transforms. "Flags" are therefore **environment variables**. (Decision confirmed: no
new dependency.)

## 3 ┬╖ Commands

**Plan-only (default, no AI, writes nothing):**

```bash
npm run diagnostic:core-v4
```

**Real generation (later, only when explicitly approved тАФ uses tokens):**

```bash
CORE_V4_APPROVE_AI=1 npm run diagnostic:core-v4
CORE_V4_APPROVE_AI=1 CORE_V4_RENDER=html,pdf npm run diagnostic:core-v4
```

**Env "flags":**

| Env | Meaning | Default |
|---|---|---|
| `CORE_V4_APPROVE_AI` | `1`/`true` to enable a real Anthropic call | unset тЖТ plan-only |
| `CORE_V4_MODEL` | primary model | `claude-sonnet-4-6` |
| `CORE_V4_FALLBACK_MODEL` | fallback model | none |
| `CORE_V4_MODE` | `sequential` \| `parallel` | `sequential` |
| `CORE_V4_OUT_DIR` | artifact directory | `outputs/core-v4-diagnostic` |
| `CORE_V4_RENDER` | comma list: `html`,`pdf` (or `all`) | none |

## 4 ┬╖ Safety

- **Plan-only is the default.** No AI call unless `CORE_V4_APPROVE_AI=1`.
- Approved-but-no-key fails clearly (`MissingApiKeyError`).
- No FreeAstroAPI (deterministic `MockAstroProvider`). No Supabase, no email, no
  checkout/order, no route. No real customer data. No persistence to Supabase.
- `pipeline.server.ts` and `system-prompt.ts` untouched; active prompt stays v3; no
  `CORE_SCHEMA_VERSION` selector. Guarded by source-level tests.

---

## 5 ┬╖ Validation stages (on an approved run)

1. Extract `report.modules.CORE` from `generateCoreV4Split()`.
2. `CoreV4Schema.safeParse()` тЖТ schema PASS/FAIL + issues.
3. `evaluateCoreV4Structure()` тЖТ missing/extra sections, wrong schema_version, etc.
4. `evaluateCoreV4Lengths()` тЖТ per-section word counts vs targets, `WARN_UNDER_TARGET`.
5. Derived checks: `before_after` pair count, executive_summary label order, closing
   pillar order, `proof_tags тЙд 5`, missing sections, under-target sections.
6. Console report via `formatValidationReport()`.

### Artifacts (approved run, under the out dir)

- `core-v4-diagnostic.raw.json` тАФ full merged report
- `core-v4-diagnostic.report.json` тАФ the CORE module
- `core-v4-diagnostic.validation.json` тАФ validation result
- `core-v4-diagnostic.html` тАФ if `CORE_V4_RENDER` includes `html`
- `core-v4-diagnostic.pdf` тАФ if `CORE_V4_RENDER` includes `pdf` (reuses `generate-v4-pdf.mjs`)

`outputs/` is gitignored тАФ artifacts are not committed.

---

## 6 ┬╖ Rendering

- **HTML:** `renderCoreV4HtmlSafe(core, clientName, clientSnapshot)` тАФ in-process.
- **PDF:** reuses `scripts/generate-v4-pdf.mjs` via `CORE_V4_PDF_IN`/`CORE_V4_PDF_OUT`
  (no second PDF engine). Same guarantees as B5.0: no browser headers/footers, no blank
  pages, paired Before/After, `Darrow Code ┬╖ NN` footer.

### Client Snapshot / Personal Orientation System (diagnostic rendering)

`buildCoreV4DiagnosticClientSnapshot(clientName, generatedCore)` assembles the snapshot
**render-only** from the generated content (professional_archetype / core_architecture /
orientation opening lines + executive_summary blocks). It does **not** mutate the core
and does **not** add `client_snapshot` / `identity_card` as a generated body key
(`CoreV4Schema.strict()` would reject those). The production bridge (computing the
snapshot from chart/numerology/BaZi metadata) remains deferred тАФ see
`docs/b5.1-core-v4-ai-prompt-hardening.md ┬з7`.
