## 3 · What was changed (staged prompt only)

File: `src/lib/ai/darrowcode_ai_system_prompt_v4_1.md` — staged, not active.

1. **Battery** (§17): `prose, 3 recharge protocols` → `prose with the 3 recharge
   methods woven into it, 1 protocol block (not three separate PROTOCOL boxes)`.
   Aligns with the accepted B5.0 rule that blocks read editorial, not schema-dumped.
2. **Editorial block rule** (§14, new line): do not stack multiple identical block
   labels in one section; at most one protocol block per section; fold extra guidance
   into prose.
3. **Before / After** (§15 + §17): replaced the grouping-implying "2 Before / 2 After"
   with "2 paired transformations" rendered **in sequence** (Before → After, then
   Before → After). The renderer already renders pairs in sequence (commit `03c0ec5`);
   the prompt must not undermine it.

## 7 · Client Snapshot / Personal Orientation System bridge — DEFERRED (B5.2/B6)

The v4 generation must **not** emit `client_snapshot` (or `identity_card`) as an 18th
section — the schema's `.strict()` rejects it and the prompt forbids it. But the PDF's
**Personal Orientation System / Client Snapshot** page needs an identity-card payload.
That payload is **template/runtime metadata**, supplied separately from the generated
body — **not** a generated body key.

Future B5.2/B6 should assemble the snapshot from:

- chart / input metadata already in the normalized `DarrowChartData`;
- computed **Sun / Moon / Ascendant** — Ascendant only if `birth_time_known = true`;
- **Life Path + Personal Year**, and name numerology (Expression / Soul Urge /
  Personality) only if the full name is available;
- **BaZi Day Master** + favorable elements only if BaZi is available;
- the **professional archetype** (generated in `professional_archetype`, or derived);
- honoring all data-availability gates (no Ascendant/houses without birth time, etc.).

No runtime bridge is implemented in B5.1.
